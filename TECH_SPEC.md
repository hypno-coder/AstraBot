## 1. ТЕХНОЛОГИЧЕСКИЙ СТЕК (СТРОГО)
* **Фреймворк бота:** `aiogram v3.4.x` (строго асинхронный).
* **Сложные UI/Диалоги:** `aiogram-dialog v2.2.0a3`.
* **БД и драйвер:** PostgreSQL 17, `asyncpg` (реляционное хранение).
* **ORM:** `SQLAlchemy v2.0.x` (современный асинхронный паттерн, typing-first).
* **Миграции:** `alembic` (в `database/`).
* **Брокер сообщений и очередей (Event-driven layer):** `NATS` v2.10+ (используется JetStream).
* **Стейт-машина (FSM Storage):** NATS KeyValue (`nats-py`), кастомная реализация.
* **Object Storage (Хранилище файлов):** NATS Object Store.
* **Микросервисный фреймворк (Взаимодействие компонентов):** `FastStream` (для обработки CPU-bound задач вне бота).
* **Интернационализация (I18N):** `fluentogram` (формат Project Fluent, `.ftl`).
* **Управление конфигурацией:** `dynaconf` (настройки в `settings.toml`, `secrets.toml`).
* **Логирование:** `structlog` (структурированные логи).
* **Пакетный менеджер:** `Poetry`.
* **ЗАПРЕЩЕНО:** Использование Redis, RabbitMQ, Celery, синхронных библиотек (requests, psycopg2 и т.д.), стандартного logging и прямых принтов.

## 2. СЛОИ АБСТРАКЦИИ (РАСПОЛОЖЕНИЕ И ИЗОЛЯЦИЯ)
* **`/database/models/`** — Слой работы с БД. Здесь располагаются декларативные модели SQLAlchemy. Логика запросов не должна протекать в конфигурацию бота. Базовый абстрактный класс (`Base`) находится в `base.py`, сущности — в `models.py`.
* **`/database/migration/`** — Скрипты и среда Alembic для управления структурой БД.
* **`/I18N/locales/`** — Слой локализации и текстов пользователя в `.ftl` формате. Запрещено хардкодить русский/английский текст в хэндлерах.
* **`/bot/handling/handlers/`** — Обработчики единичных событий бота (например, `start.py`, `get_user.py`). 
* **`/bot/handling/dialogs/`** — Описание сложных UI-стейтов на базе `aiogram-dialog` (экраны, виджеты, пагинация).
* **`/bot/handling/states/`** — Описание графов состояний (FSM) для бота.
* **`/bot/handling/middlewares/`** — Промежуточный слой (Interceptors). Отвечает за инъекцию зависимостей, управление сессиями БД и получение языковых профилей.
* **`/bot/nats_storage/`** — Слой хранения FSM-стейтов (персистентное хранилище на базе NATS KV).
* **`/img-converter/`** (или аналогичные сервисы) — Автономные фоновые worker-ы. Изолированные микросервисы на FastStream, не зависящие от aiogram. Обрабатывают бинарные и CPU-bound задачи.

## 3. ПАТТЕРНЫ ПРОЕКТИРОВАНИЯ
* **Dependency Injection (DI):** Внедрение зависимостей происходит глобально через Middlewares. Хэндлеры не создают подключения к БД, не читают файлы конфига напрямую — все передается через аргументы функции (`**kwargs` в `aiogram`, например: `i18n: TranslatorRunner`, `dialog_manager`).
* **State Machine Pattern (FSM):** Контекст пользователя хранится в `NATSFSMStorage` строго вне оперативной памяти бота.
* **Event-Driven / Publisher-Subscriber Pattern:** Вся блокирующая (CPU / I/O длительная) бизнес-логика выводится из цикла бота. Бот сериализует задачу (`Task`), пушит в NATS -> Worker подхватывает, обрабатывает и возвращает ответ.
* **Claim Check Pattern:** Вместо передачи бинарных данных (картинок/документов) через шину брокера, объект загружается в NATS Object Store, а в брокер передается лишь идентификатор (`img_uuid`).
* **Active Record / Repository Model:** Использование `relationship(lazy="selectin")` в SQLAlchemy 2.0 для предзагрузки зависимых сущностей во избежание N+1 запросов.

## 4. ПРАВИЛА КОД-СТАЙЛА И РОУТИНГА
* **Роутинг:** Запрещено добавлять хэндлеры напрямую в `Dispatcher` (dp). Каждая предметная область (модуль) имеет свой `Router()`. Регистрация всех роутеров строго консолидирована в `/bot/handling/schema.py` (`dp.include_router(...)`).
* **Типизация (PEP 484):** 100% покрытие Type Hints (Python 3.10+). Исключительно явный синтаксис (например: `Mapped[int]`).
* **Сессии БД:** Управление сессиями инвертировано. `DatabaseMiddleware` инициализирует транзакцию, прокидывает абстракцию в хэндлер и делает commit/rollback после завершения хэндлера. Явные `session.commit()` внутри хэндлеров запрещены (за исключением сложных долгих транзакций).
* **Dialogs vs Handlers:** Простые текстовые команды (`/start`) перехватываются классическими Router'ами в `/handlers`. Вся навигация, пагинация, сбор данных (шаги) перехватываются в `/dialogs` с помощью `DialogManager`. Смешивание логик не допускается.
* **Logging:** Полное отстранение от стандартного `print()` и `logging`. Разрешено использовать только `import structlog`. Любые события сопровождаются JSON/текст логом: `logger.info()`.
